## Implementation Steps

> **Ordering constraint (critical, from Codex review).** The three regex-and-fence tasks below MUST land in the order Task 1 â†’ Task 2 â†’ Task 3, never the other way. Rationale: if the forgery regex is tightened (Task 3) while `USER QUESTION` and `CANDIDATES` fences are still plain-ASCII (current state of `pkg/prompt/expert.go` and `pkg/debate/vote.go`), then forged `=== END USER QUESTION ===` / `=== END CANDIDATES ===` lines would slip past the tightened `[nonce-<hex>]`-shape requirement and could prematurely close downstream fenced sections â€” the exact break-out ADR-0011 aims to eliminate. Doing fence-tagging FIRST keeps the invariant safe: broad regex + more nonce-tagged fences is strictly no worse than today; only once every real fence is nonce-bearing do we tighten the regex to match.
